# 二次开发启动说明（烽火笔锋）

## 1. 本次对话结论（摘要）

本次讨论将项目方向从“通用 AI 剧情互动”收敛为：

- 项目名称：**烽火笔锋**
- 目标形态：**抗战历史情境下的战地记者决策与沟通能力训练系统**
- 核心时间轴：**1937-07-07 至 1945-09-02**
- 核心方法：**历史锚点约束 + 二次创作剧本 + KT（知识追踪）能力建模**

我们明确了不做“娱乐化改史”，而是做“历史情境中的高风险沟通训练与评估”。

---

## 2. 已达成的产品与技术定位

### 2.1 产品定位

这是一个“可训练、可评估、可成长”的教育型 AI 系统，而不是纯剧情游戏。

用户角色：战地记者  
训练目标：在高压信息环境中提升事实核验、风险沟通、伦理边界等能力。

### 2.2 三层结构

1. **史实层（不可改）**：固定历史锚点、关键事件和结论  
2. **创作层（可演绎）**：人物关系、现场细节、对话冲突、决策情境  
3. **训练层（可评估）**：每轮决策映射技能评分并更新 KT 状态

---

## 3. 剧情走向与结局设计原则（已确定）

玩家每次决策将同时影响两类状态：

1. **剧情状态 S**：可信度、准确性、公众恐慌、线人安全、编辑部信任等  
2. **能力状态 K（KT）**：K1-K8 八维能力掌握度

并通过以下机制推进：

- 决策 -> 状态更新 -> 下一场景选择 -> 累计状态判定结局

结局不是“改写抗战胜负”，而是“记者个人与传播后果结局”，例如：

- 史笔如铁
- 艰难守真
- 逆风修正
- 失真扩散
- 代价沉重

---

## 4. 关键难点与应对策略（开发期）

1. **历史真实性约束难**  
解决：时间轴与事件硬约束，LLM 输出前后双重校验，不通过则回退模板。

2. **LLM 评分波动**  
解决：Rubric 结构化评分 + JSON Schema 输出 + 规则引擎二次校准。

3. **KT 冷启动数据不足**  
解决：先上 BKT（专家先验参数），后续样本足够再引入 DKT。

4. **训练与剧情平衡难**  
解决：每个场景固定 2-3 个高价值决策点，避免分支爆炸。

5. **现场演示稳定性风险**  
解决：准备离线演示模式与预置剧本回放。

---

## 5. 二次开发仓库的建议边界

建议新仓库只保留“训练系统主干”，剥离与当前项目主线无关的模块，避免历史包袱过重。

### 5.1 建议迁移的核心能力

- 场景推进与多轮交互框架（可复用现有 game service 思路）
- 向量检索与记忆机制（用于复盘、相似案例检索）
- LLM 多 provider 接入能力
- TTS 语音交互能力
- 前端交互页骨架与会话恢复机制

### 5.2 建议新建的核心模块

- `history/`：历史时间轴与事件约束
- `script/`：剧本与决策点数据
- `kt/`：BKT/DKT/评估策略
- `training/`：训练会话、评分、报告

## 7. 接下来工作计划（建议 4 阶段）

## 阶段 A：仓库初始化（1 周）

1. 新仓库创建与基础工程搭建  
2. 固定历史时间轴与事件约束模块落地  
3. 首个剧本场景（卢沟桥）数据模型定稿

## 阶段 B：训练闭环 MVP（2-3 周）

1. 单场景多轮决策流程跑通  
2. LLM 结构化评分 + BKT 更新  
3. 前端雷达图与本轮反馈页上线

## 阶段 C：多场景与结局（2 周）

1. 扩展到 4-6 个核心场景  
2. 自适应推题策略上线  
3. 结局判定与训练报告生成

## 阶段 D：验证与答辩（1-2 周）

1. 前测/后测实验与指标统计  
2. 演示脚本、演示数据、视频录制  
3. 文档整理（应用场景、技术方案、测试结果、部署说明）

---

## 8. 第一批必须交付（最小可用）

1. 1 个完整历史场景（含 3 个决策点）  
2. 1 套八维能力评分 Rubric  
3. BKT 状态更新与雷达图  
4. 1 个可解释结局页（含证据链）

## 9. 与当前仓库的关系

- 当前仓库可作为“技术能力来源库”（LLM、TTS、向量库、交互流程）。
- 新仓库作为“产品化与比赛交付仓库”，建议避免直接复制全部历史文件。
- 推荐策略：**按模块迁移 + 边迁移边重构 + 保持文档先行**。

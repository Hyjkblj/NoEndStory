# 事件推进器详细分析

## 一、整体架构

### 核心组件
1. **StoryEngine（故事引擎）**：主控制器，管理事件流程
2. **EventGenerator（事件生成器）**：生成事件内容和对话
3. **VectorDatabase（向量数据库）**：存储和检索历史事件
4. **DatabaseManager（数据库管理器）**：管理角色状态和属性

### 关键状态变量
```python
self.current_event_count = 0      # 当前事件计数（0=开头，1-15=中间，>15=结尾）
self.current_event = None         # 当前事件信息（包含story_background、scene等）
self.dialogue_history = []        # 当前事件的对话历史（角色对话+玩家选择）
self.min_dialogue_rounds = 4      # 最少对话轮数（每轮=角色对话+玩家选择）
self.current_scene = 'classroom'  # 当前场景（小场景ID）
self.previous_event_contexts = [] # 已发生的事件上下文（避免重复）
```

---

## 二、游戏流程总览

```
初始化游戏
    ↓
【阶段1】开头事件（初遇）
    ↓
【阶段2】中间事件循环（1-15个事件）
    ├─ 每个事件包含4+轮对话
    ├─ 每轮对话：角色对话 → 玩家选择 → 保存到向量DB
    └─ 事件结束 → 保存完整事件到向量DB → 推演下一个事件
    ↓
【阶段3】结尾事件（根据状态值判定结局）
    ↓
游戏结束
```

## 三、详细流程分析

### 3.1 初始化阶段

**调用路径**：`GameService.initialize_story()` → `StoryEngine.get_opening_event()`

**流程**：
1. 玩家选择大场景（如'school'）
2. 从大场景的`opening_events`列表中随机选择一个初遇事件
3. 获取事件对应的小场景ID（如'classroom'）
4. 调用`EventGenerator.generate_event()`生成故事背景
5. 设置`current_event_count = 1`，`current_scene = 小场景ID`
6. 返回事件信息（包含story_background、scene等）

**关键代码**：
```python
# 从10个初遇事件中随机选择
selected_event = random.choice(opening_events)

# 生成故事背景（使用AI，基于角色信息和事件描述）
event = self.event_generator.generate_event(
    character_id=character_id,
    event_id=selected_event['id'],
    event_context=selected_event['description'],
    previous_contexts=self.previous_event_contexts
)
```

---

### 3.2 对话轮次机制

**调用路径**：`GameService.process_input()` → `StoryEngine.get_next_dialogue_round()`

**流程**：
1. 玩家做出选择（选项ID或自由输入）
2. 调用`process_player_choice()`：
   - 记录玩家选择到`dialogue_history`
   - 更新角色状态值（如果选项有state_changes）
3. 调用`save_dialogue_round_to_vector_db()`：
   - 将本轮对话（角色对话+玩家选择）保存到向量数据库
4. 检查`should_continue_dialogue()`：
   - 如果`dialogue_history`长度 < 4（即对话轮次 < 2），继续对话
   - 否则，结束当前事件

**对话轮次计算**：
```python
dialogue_round = (len(self.dialogue_history) // 2) + 1
# dialogue_history格式：[角色对话, 玩家选择, 角色对话, 玩家选择, ...]
# 每2个元素 = 1轮对话
```

**生成对话**：
```python
# 从向量数据库检索最近的对话（用于连贯性）
recent_dialogues = vector_db.search_recent_dialogues(...)

# 生成角色对话（基于故事背景、角色信息、状态值、历史对话）
character_dialogue = event_generator.generate_character_dialogue(...)

# 生成玩家选项（基于角色对话、故事背景、历史对话）
player_options = event_generator.generate_player_options(...)
```

---

### 3.3 中间事件循环

**调用路径**：`GameService.process_input()` → `StoryEngine.get_next_event()` → `StoryEngine.get_middle_event()`

**事件计数逻辑**：
```python
def get_next_event(self, character_id: int, scene_id: str = 'school'):
    if self.current_event_count == 0:
        return self.get_opening_event(...)  # 开头事件
    elif self.current_event_count <= config.GAME_CONFIG['max_events']:  # max_events = 15
        return self.get_middle_event(...)   # 中间事件
    else:
        return self.get_ending_event(...)   # 结尾事件
```

**中间事件生成流程**：

#### 步骤1：检索历史事件
```python
previous_events = vector_db.search_similar_events(
    character_id=character_id,
    query="场景 地点 位置 去 到 在",  # 重点关注场景相关信息
    n_results=10
)
```

#### 步骤2：推演下一个场景
调用`_infer_next_scene_from_history()`：
1. **AI推演**（优先）：
   - 分析历史事件文本，提取场景关键词
   - 如果历史事件明确提到"去咖啡厅"、"明天一起学习"等，选择对应场景
   - 如果暗示了场景（如"聊到要去咖啡厅"），选择该场景
   - 如果没有明确提到，选择与当前场景有自然关联的场景
2. **关键词匹配**（备选）：
   - 从历史事件文本中直接匹配场景名称和关键词
   - 计算匹配分数，选择得分最高的场景
3. **随机选择**（最后备选）：
   - 从当前大场景的小场景列表中随机选择
   - 70%概率切换场景，30%概率保持当前场景

#### 步骤3：生成事件上下文
调用`_generate_event_context()`：
1. **判断剧情阶段**：
   - 事件1-2：初期（自然、日常互动）
   - 事件3-5：发展期（更深入的互动）
   - 事件6+：深入期（更私人的互动）
2. **构建历史事件摘要**：
   - 从向量数据库获取最近5个历史事件
   - 从`previous_event_contexts`获取已发生事件摘要
3. **AI生成事件描述**（40-60字）：
   - 基于历史事件自然推演
   - 确保连续性和合理性
   - 符合剧情阶段要求

#### 步骤4：生成完整事件
```python
event = self.event_generator.generate_event(
    character_id=character_id,
    event_id=f"middle_event_{event_number}",
    event_context=event_context,  # AI生成的事件描述
    previous_contexts=self.previous_event_contexts
)
# 这会调用AI生成story_background（详细的故事背景文本）
```

#### 步骤5：场景切换处理
如果场景切换了：
1. 生成新场景图片
2. 合成场景图+角色图
3. 更新`current_scene`

#### 步骤6：更新状态
```python
self.current_event_count = event_number + 1
self.current_event = event
self.dialogue_history = []  # 重置对话历史
self.previous_event_contexts.append(event_context[:150])  # 记录事件上下文
```

---

### 3.4 事件结束处理

**触发条件**：`should_continue_dialogue()`返回False（对话轮次 >= 4）

**流程**：
1. 调用`save_event_to_vector_db()`：
   - 将当前事件的所有对话保存到向量数据库
   - 包含：story_background、完整对话文本、metadata（场景、标题等）
2. 检查`is_game_finished()`：
   - 如果`current_event_count > max_events`（15），进入结尾
   - 否则，获取下一个中间事件

**判断逻辑**：
```python
def is_game_finished(self) -> bool:
    return self.current_event_count > config.GAME_CONFIG['max_events']
```

---

### 3.5 结尾事件

**调用路径**：`StoryEngine.get_ending_event()`

**流程**：
1. 获取角色状态值：
   - `favorability`（好感度）
   - `trust`（信任度）
   - `hostility`（敌意值）
2. 根据状态值判定结局类型：
   ```python
   if favorability > 70 and trust > 60 and hostility < 20:
       ending_type = "happy_ending"  # 美好结局
   elif favorability < 30 or hostility > 50:
       ending_type = "bad_ending"   # 坏结局
   elif trust > 50 and favorability > 40:
       ending_type = "neutral_ending"  # 中性结局
   else:
       ending_type = "open_ending"  # 开放结局
   ```
3. 生成结尾事件（包含故事背景）
4. 进行结尾对话（同样支持多轮对话）

## 四、数据存储机制

### 4.1 向量数据库存储

**存储时机**：
1. **每轮对话后**：`save_dialogue_round_to_vector_db()`
   - 存储内容：角色对话 + 玩家选择
   - 用途：用于生成下一轮对话的连贯性
2. **每个事件结束后**：`save_event_to_vector_db()`
   - 存储内容：完整事件（story_background + 所有对话）
   - 用途：用于推演下一个事件和场景

**存储格式**：
```python
{
    'character_id': character_id,
    'event_id': event_id,
    'story_text': story_background,  # 故事背景
    'dialogue_text': dialogue_text,   # 完整对话
    'metadata': {
        'event_context': event_context,
        'scene': scene_id,
        'title': event_title,
        ...
    }
}
```

### 4.2 状态值更新

**更新时机**：玩家选择选项时（如果选项有`state_changes`）

**更新方式**：
```python
if choice['type'] != 'neutral' and choice.get('state_changes'):
    self.db_manager.update_character_states(
        character_id=character_id,
        state_changes=choice['state_changes']  # 例如：{'favorability': +5}
    )
```

## 五、场景切换逻辑详解

### 5.1 场景推演优先级

1. **明确提及**（最高优先级）：
   - 历史事件中明确提到"去咖啡厅"、"到教室"等
   - 必须选择该场景

2. **暗示场景**（次优先级）：
   - 历史事件中暗示"聊到要去咖啡厅"、"说好一起学习"
   - 选择对应场景

3. **自然过渡**（第三优先级）：
   - 选择与当前场景有自然关联的场景
   - 例如：教室→图书馆（学习相关）、教室→食堂（时间顺序）

4. **随机选择**（最后备选）：
   - 从当前大场景的小场景列表中随机选择
   - 70%概率切换，30%概率保持

### 5.2 场景切换时的图片处理

如果场景切换了：
1. 生成新场景图片（使用场景描述和关键词）
2. 获取角色最新图片
3. 合成场景图+角色图
4. 返回合成图片URL给前端

## 六、剧情发展阶段控制

### 6.1 阶段划分

- **初期**（事件1-2）：自然、日常互动，保持真实感
- **发展期**（事件3-5）：更深入的互动，关系慢慢熟悉
- **深入期**（事件6+）：更私人的互动，关系逐渐加深

### 6.2 循序渐进机制

1. **关系发展限制**：
   - 如果之前只是点头之交，现在不能突然变成亲密关系
   - 关系发展要符合逻辑，一步步来

2. **事件连续性**：
   - 新事件必须与历史事件有明确的因果关系或时间顺序
   - 如果历史事件提到"明天一起去图书馆"，这次事件就可以在图书馆发生

3. **合理性检查**：
   - 事件要符合学生日常生活的逻辑
   - 避免过于戏剧化、不切实际的情节

## 九、总结

### 核心机制
1. **事件驱动**：游戏以事件为单位推进，每个事件包含多轮对话
2. **向量数据库记忆**：所有历史事件和对话都保存到向量数据库，用于推演
3. **场景智能推演**：基于历史事件自动推演下一个场景，确保逻辑连贯
4. **循序渐进**：剧情发展分阶段，关系发展符合逻辑
5. **状态值影响**：玩家选择会影响角色状态值，最终影响结局

### 数据流
```
玩家选择 → 更新状态值 → 保存对话到向量DB
    ↓
检查对话轮次 → 继续对话 OR 结束事件
    ↓
事件结束 → 保存完整事件到向量DB
    ↓
推演下一个场景 → 生成事件上下文 → 生成故事背景
    ↓
开始新事件的对话循环
```

### 关键设计
- **避免重复**：通过`previous_event_contexts`和向量数据库检索避免重复事件
- **保持连贯**：通过历史事件检索和AI生成确保剧情连续性
- **自然推进**：通过场景推演和阶段控制确保剧情自然发展
- **状态驱动**：通过状态值影响对话生成和结局判定

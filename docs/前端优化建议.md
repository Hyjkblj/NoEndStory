# 前端项目优化建议

基于对 `frontend/` 的阅读与分析，从结构、性能、可维护性、API 使用、状态管理等方面给出优化建议，并按优先级排序。

---

## 一、高优先级

### 1. 健康检查接口路径与代理

**问题**：`src/services/api.ts` 中 `checkServerHealth` 使用 `axios.get('/health')`（未使用带 `baseURL: '/api'` 的 api 实例）。开发环境下 Vite 只代理了 `/api` 和 `/static`，请求会发往 `http://localhost:3000/health`，得到 404，导致“无法连接服务器”误判。

**建议**：
- 方案 A：在 `vite.config.ts` 的 `server.proxy` 中增加 `/health` 转发到 `http://localhost:8000`。
- 方案 B：健康检查使用与后端一致的完整地址，例如 `axios.get('http://localhost:8000/health')` 或通过环境变量配置 `VITE_API_ORIGIN`，开发时用该变量拼出 `/health` 的完整 URL。

**参考**：后端健康检查为 `GET http://localhost:8000/health`（见 `backend/api/app.py`）。

---

### 2. Game.tsx 体量过大，拆分子逻辑与子组件

**问题**：`src/pages/Game.tsx` 单文件约 900+ 行，集中了初始化、存档、选项处理、场景/图片/TTS、转场、UI 等全部逻辑，难以维护和单测。

**建议**：
- **状态与数据**：将 `threadId`、`characterId`、`messages`、`currentOptions`、场景/图片相关状态等收敛到自定义 Hook（如 `useGameState`），或使用 Context + reducer 管理“游戏状态”。
- **初始化与恢复**：将“恢复存档 / 从 FirstMeeting 进入 / 新游戏初始化”等逻辑抽到 `useGameInit` 或独立模块，通过回调或状态更新与 UI 交互。
- **子组件**：
  - 场景/合成图/占位符展示 → `GameSceneBackground`
  - 对话框 + 选项按钮 → `GameDialogue`
  - 转场 → 已有 `SceneTransition`，可保持
- **工具函数**：`getSceneName`、`getCharacterImageFromStorage`、`loadCharacterImageFromAPI`、`setCharacterImage` 等移入 `utils/game.ts` 或 `config/scenes.ts`（场景名映射可与 `scenes.ts` 统一），便于复用和测试。

---

### 3. 统一 API 响应类型与拦截器返回值

**问题**：拦截器里 `return response.data`，导致调用方拿到的有时是“完整响应体”（如 `{ code, message, data }`），有时是 `data`。代码中大量出现 `(response as any)?.data || response` 的兼容写法，类型不清晰，易出错。

**建议**：
- 约定：拦截器统一只返回 `response.data`（即后端 `data` 字段），或统一返回 `{ data: response.data, status: response.status }`，二选一并全文统一。
- 为各接口定义明确类型（如 `CreateCharacterResponse`、`GameInitResponse`），在 api 层做类型断言或泛型，避免在页面里 `as any`。
- 若保留“兼容旧代码”的写法，集中放在 api 层的一个 `normalizeResponse(response)` 中，并在文档中注明废弃时间，逐步改为单一约定。

---

### 4. sessionStorage / localStorage 抽象与类型安全

**问题**：多处直接使用 `sessionStorage.getItem('characterData')`、`localStorage.getItem('gameSave_xxx')` 等，键名和结构分散在各页面，易拼错、难重构，且没有统一的类型和解析错误处理。

**建议**：
- 增加一层存储抽象，例如 `src/storage/gameStorage.ts`：
  - 集中定义 key 常量（如 `CHARACTER_DATA_KEY`、`GAME_SAVE_KEY`）。
  - 提供 `getCharacterData()`、`setCharacterData()`、`getGameSave(threadId)`、`setGameSave(...)` 等函数，内部做 `JSON.parse` + try/catch，返回类型化结果或 null。
- 对 `characterData`、`GameSave` 等结构在 `types/` 中定义接口，存储层只依赖这些类型，避免各处手写 any。

---

## 二、中优先级

### 5. 路由与路径常量

**问题**：路由 path 为字符串字面量（如 `'firststep'`、`'game'`），在 `navigate('/firststep')` 等处重复书写，易 typo，且修改路径时需全局搜索。

**建议**：在 `src/router/` 或 `src/config/routes.ts` 中定义路径常量及 `buildPath`（若将来有动态参数），例如：

```ts
export const ROUTES = {
  HOME: '/',
  FIRST_STEP: '/firststep',
  CHARACTER_SETTING: '/charactersetting',
  CHARACTER_SELECTION: '/characterselection',
  FIRST_MEETING: '/firstmeeting',
  GAME: '/game',
} as const;
```

路由表与 `navigate()` 均使用 `ROUTES.xxx`。

---

### 6. 401 跳转与“登录”态

**问题**：api 响应拦截器中 401 会 `localStorage.removeItem('token')` 并 `window.location.href = '/login'`，但当前项目未见登录页或登录流程，可能导致用户被重定向到不存在的页面。

**建议**：若暂无登录功能，可先将 401 处理改为仅提示“未授权”并可选跳转到首页或 FirstStep；若后续接入登录，再统一在路由或 Layout 中做鉴权与重定向。

---

### 7. 场景名称映射重复

**问题**：`Game.tsx` 内 `getSceneName` 维护了一套 `sceneId -> 中文名` 的映射，与 `config/scenes.ts` 中的 `SCENE_CONFIGS` 部分重叠且可能不一致。

**建议**：在 `config/scenes.ts` 中导出 `getSceneNameById(sceneId: string): string`，实现上基于 `SCENE_CONFIGS` 查找，找不到时返回 `sceneId`。Game 及所有使用场景名的地方统一使用该函数，避免两处维护。

---

### 8. 控制台日志与生产环境

**问题**：多处 `console.log` / `console.warn` / `console.error`（如 API 请求参数、响应、sessionStorage 检查等），生产环境会暴露内部信息并增加噪音。

**建议**：使用简单日志工具（如 `src/utils/logger.ts`），根据 `import.meta.env.DEV` 或 `VITE_LOG_LEVEL` 决定是否输出；或保留 `console.error` 用于真实错误，将调试类 `console.log` 改为 `logger.debug` 并在生产关闭。

---

### 9. 类型定义集中与复用

**问题**：`Game.tsx` 内定义了本地 `Message`、`PlayerOption`、`GameSave` 等接口，与 `types/index.ts` 中的 `Message` 等可能重复或不一致；api 层部分接口已定义 request/response 类型，但未在类型文件中统一导出。

**建议**：将游戏相关类型（含消息、选项、存档）收拢到 `types/game.ts` 或 `types/index.ts`，api 的 request/response 类型可继续在 `api.ts` 或单独 `types/api.ts` 中定义并导出，页面只做引用，避免重复定义。

---

## 三、低优先级与长期改进

### 10. Hooks 与 Context 使用

**问题**：`hooks/index.ts`、`contexts/index.ts` 目前为空占位，游戏状态、角色数据等均通过 props 或 sessionStorage 在页面间传递，可测试性和复用性有限。

**建议**：在拆分 Game 状态（见建议 2）时，引入 `useGameState`、`useGameInit` 等 Hook；若存在多页面需要的“当前角色”“当前存档”等，可考虑 `GameContext` 或 `CharacterContext`，由 Layout 或路由级 Provider 提供，减少 sessionStorage 直接读写。

---

### 11. 样式与主题

**问题**：部分页面（如 Home）大量内联 style，Layout 与 Game 使用 CSS 文件，风格不统一；主题色（如橙色、金色）分散在多处。

**建议**：将可复用样式（如标题、主按钮、背景遮罩）抽成 CSS Modules 或全局 class；主题色、字体等放入 `config/theme.ts` 或 CSS 变量，与 antd 的 `themeConfig` 保持一致，便于后续换肤或统一调整。

---

### 12. 加载与错误 UI 统一

**问题**：Loading 已有统一组件（LoadingScreen），但 Game 内加载为自定义 `game-loading-overlay` + Spin，错误态多为 `message.error`，缺少统一的“全屏错误页”或“重试”能力。

**建议**：对关键流程（如初始化故事、处理选项）可封装“带 loading + 错误 + 重试”的边界组件或 Hook（如 `useAsyncOperation`），错误时展示统一错误区域和重试按钮，减少各页面重复逻辑。

---

### 13. 可访问性与语义化

**问题**：未看到对焦点管理、键盘操作、ARIA 的明显处理；部分仅用 div + style 的区块可用语义化标签（如 `main`、`section`、`nav`）以利无障碍与 SEO。

**建议**：在关键交互（如选项按钮、对话框）上增加焦点与键盘支持；大块结构用语义化标签；重要图标或装饰图加 `alt` 或 `aria-hidden` 等，按需逐步完善。

---

### 14. 测试与 E2E

**问题**：未见前端单测或 E2E 配置。

**建议**：在时间允许的情况下，优先为 api 层、存储层和核心 Hook（如未来的 `useGameState`）编写单元测试；对“开始游戏 → 选角色 → 进入游戏 → 选选项”等关键路径增加少量 E2E（如 Playwright），防止重构时回归。

---

## 四、简要检查清单

| 项 | 说明 |
|----|------|
| 健康检查 | 修正 `/health` 的请求目标或代理，确保开发/生产都能正确检测后端 |
| Game 拆分 | 状态 Hook、初始化逻辑、场景/对话子组件、工具函数抽出 |
| API 响应 | 统一拦截器返回约定与类型，减少 `as any` 与重复兼容写法 |
| 存储层 | 抽象 session/localStorage，统一 key 与类型，集中解析与错误处理 |
| 路由常量 | 路径常量集中定义，避免硬编码字符串 |
| 401 与登录 | 根据是否具备登录功能，调整 401 行为与重定向 |
| 场景名 | 单一数据源（scenes 配置），统一 `getSceneNameById` |
| 日志 | 区分开发/生产，收敛调试日志 |
| 类型 | 游戏与 API 类型集中定义与导出，避免重复 |
| Hooks/Context | 随 Game 拆分引入状态与初始化 Hook，必要时加 Context |
| 样式与主题 | 统一主题变量与复用样式，减少内联 |
| 错误与重试 | 关键流程统一错误展示与重试能力 |
| 无障碍与测试 | 按需增加焦点/键盘/语义化；为 api、存储、核心 Hook 及主流程加测 |

---

以上建议优先落地“健康检查”“Game 拆分”“API 响应统一”“存储抽象”和“路由常量”，能显著提升可维护性与可测性，并减少线上误判与潜在 bug。若需要，我可以按某一项给出具体修改示例（含文件路径和代码片段）。

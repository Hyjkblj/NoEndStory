# No End Story - 数据量分析与估算

## 一、数据量估算基础

### 1.1 项目特性

- **游戏类型**：无限剧情流游戏
- **核心功能**：AI 生成剧情 + 图像
- **存储需求**：结构化数据 + 向量数据 + 图像数据

### 1.2 用户行为假设

基于无限剧情游戏的特点：

- **平均游戏时长**：单次 30-60 分钟
- **平均对话轮次**：每次游戏 20-50 轮对话
- **活跃用户频率**：每周 2-3 次
- **游戏保留率**：活跃用户会保留多个 Thread（存档）

## 二、分阶段数据量估算

### 2.1 MVP 阶段（0-3 个月）

#### 用户规模
- **用户数**：100-500 人
- **活跃用户**：50-200 人（每周至少 1 次）
- **总 Thread 数**：200-1000 个

#### 数据结构量估算

**1. 用户表 (users)**
```
用户数：500
每条记录：~200 bytes（UUID + username + timestamp）
总数据量：500 × 200 bytes = 100 KB
```

**2. 线程表 (threads)**
```
Thread 数：1000
每条记录：~150 bytes（UUID × 2 + openai_thread_id + timestamp）
总数据量：1000 × 150 bytes = 150 KB
```

**3. 剧情状态表 (story_states)**
```
状态数：1000（每个 Thread 一个状态）
每条记录：~2 KB（UUID + scene + JSONB 数据）
总数据量：1000 × 2 KB = 2 MB
```

**4. 对话历史表 (conversations)**
```
平均每个 Thread：30 轮对话
总对话数：1000 × 30 = 30,000 条
每条记录：~1 KB（UUID + role + content + metadata）
总数据量：30,000 × 1 KB = 30 MB
```

**5. 图像缓存表 (image_cache)**
```
假设缓存 1000 张图像
每条记录：~500 bytes（UUID + hash + URL + metadata）
总数据量：1000 × 500 bytes = 500 KB
```

**结构化数据小计**：~32.75 MB

#### 向量数据量估算

**剧情记忆向量表 (story_memories)**
```
平均每个 Thread：50 条记忆（每次对话生成 1-2 条记忆）
总记忆数：1000 × 50 = 50,000 条
每条记录：
  - 向量：1536 维度 × 4 bytes = 6,144 bytes
  - 元数据：~500 bytes（content + metadata）
  - 总计：~6.5 KB
总数据量：50,000 × 6.5 KB = 325 MB
```

**向量数据小计**：~325 MB

#### 图像数据量估算

**存储方式**：图像存储在对象存储（S3/OSS）或本地文件系统，数据库只存储 URL

```
假设缓存 1000 张图像
每张图像：~500 KB（DALL·E 3 标准质量）
总数据量：1000 × 500 KB = 500 MB
```

**图像数据小计**：~500 MB

#### MVP 阶段总数据量

| 数据类型 | 数据量 |
|---------|--------|
| **结构化数据** | ~33 MB |
| **向量数据** | ~325 MB |
| **图像数据** | ~500 MB |
| **总计** | **~858 MB（约 0.86 GB）** |

## 三、详细数据量分析

### 3.1 单用户数据量

#### 活跃用户（典型场景）

**假设**：一个活跃用户每周玩 3 次，每次 40 分钟，30 轮对话

**每周数据量**：
- 对话记录：3 次 × 30 轮 × 2（用户+AI）× 1 KB = 180 KB
- 记忆向量：3 次 × 30 轮 × 1.5 条 × 6.5 KB = 877 KB
- 图像：3 次 × 10 张 × 500 KB = 15 MB
- 状态数据：3 次 × 2 KB = 6 KB

**每周小计**：~16 MB/用户

**每月数据量**：~64 MB/用户

**每年数据量**：~768 MB/用户（约 0.8 GB/用户）

#### 不活跃用户（长期留存）

- 可能有多个 Thread（存档）
- 但很少更新
- 数据量：~10-50 MB/用户

## 七、数据优化策略

### 7.1 数据清理策略

**1. 对话历史归档**
```
策略：超过 6 个月未活跃的 Thread 归档
效果：减少 30-50% 数据量
```

**2. 向量数据清理**
```
策略：删除重要性评分 < 0.3 的记忆
效果：减少 20-30% 向量数据
```

**3. 图像缓存清理**
```
策略：超过 1 年未访问的图像删除
效果：减少 20-40% 图像数据
```

### 7.2 数据压缩

**1. 向量压缩**
- 使用量化技术（16-bit 代替 32-bit）
- 可减少 50% 存储空间
- 对精度影响 < 5%

**2. JSON 压缩**
- PostgreSQL JSONB 自动压缩
- 可减少 30-50% 存储空间

## 八、总结

### 8.1 关键数据量指标

| 阶段 | 用户数 | Thread 数 | 向量数 | 总数据量 | 数据库容量 |
|------|--------|-----------|--------|---------|-----------|
| **MVP（0-3月）** | 500 | 1,000 | 5 万 | **~0.86 GB** | **1 GB** |
| **初期（3-12月）** | 5,000 | 20,000 | 100 万 | **~10 GB** | **20 GB** |
| **增长（12-24月）** | 20,000 | 200,000 | 500 万 | **~100 GB** | **200 GB** |

### 8.2 技术选型建议

**基于数据量选择**：

1. **MVP 阶段（< 10 万向量）**
   - ✅ PostgreSQL + pgvector：性能优秀，架构统一
   - ✅ Chroma：性能足够，开发快速
   - 两者都适合

2. **初期阶段（10万-100万向量）**
   - ✅ PostgreSQL + pgvector：性能良好，架构统一（推荐）
   - ✅ Chroma：性能良好
   - 两者都适合，推荐 pgvector（统一架构）

3. **增长阶段（100万-500万向量）**
   - ⚠️ PostgreSQL + pgvector：性能开始下降，需优化
   - ✅ Chroma：性能仍然良好
   - ✅ 可考虑迁移到专用向量数据库（Pinecone、Weaviate）

### 8.3 容量规划建议

**MVP 阶段**：
- 数据库：1-2 GB（预留 2 倍空间）
- 图像存储：2 GB
- **总计：3-4 GB**

**初期阶段**：
- 数据库：20-30 GB（预留 1.5 倍空间）
- 图像存储：10 GB
- **总计：30-40 GB**

**增长阶段**：
- 数据库：200-300 GB（预留 1.5 倍空间）
- 图像存储：50 GB
- **总计：250-350 GB**

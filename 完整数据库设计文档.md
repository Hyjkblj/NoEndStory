# 完整数据库设计文档

## 📊 数据库架构总览

本项目使用**双数据库架构**：
1. **PostgreSQL**：存储结构化数据（角色信息、状态值）
2. **ChromaDB（向量数据库）**：存储非结构化文本数据（事件、对话），支持语义检索

---

## 🗄️ 第一部分：PostgreSQL数据库设计

### 数据库配置
- **类型**：PostgreSQL
- **连接配置**：`backend/config.py`
- **ORM框架**：SQLAlchemy
- **数据库名**：`noendstory`（默认）

### 表结构设计

#### 1. `characters` 表（角色主表）

**用途**：存储角色的基本信息

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | INTEGER | PK, AUTO_INCREMENT | 角色ID（主键） |
| `name` | VARCHAR(100) | NOT NULL | 角色名称 |
| `gender` | VARCHAR(20) | NOT NULL | 性别 |
| `appearance` | TEXT | NOT NULL | 外观描述 |
| `personality` | TEXT | NOT NULL | 性格描述 |
| `scene_id` | VARCHAR(50) | DEFAULT 'school' | 初始场景ID |
| `created_at` | DATETIME | DEFAULT NOW() | 创建时间 |
| `updated_at` | DATETIME | DEFAULT NOW() | 更新时间 |

**关联关系**：
- 一对多：`Character` → `CharacterAttribute`（角色决定因素）
- 一对一：`Character` → `CharacterState`（角色状态值）

---

#### 2. `character_attributes` 表（角色决定因素表）

**用途**：存储角色的各种决定因素（性格、背景、价值观等）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | INTEGER | PK, AUTO_INCREMENT | 记录ID |
| `character_id` | INTEGER | FK → characters.id | 角色ID（外键） |
| `attribute_type` | VARCHAR(50) | NOT NULL | 属性类型 |
| `attribute_value` | TEXT | NOT NULL | 属性值 |
| `created_at` | DATETIME | DEFAULT NOW() | 创建时间 |

**属性类型（`attribute_type`）**：
- `家庭背景`：12种选项（城市中产、农村出身、单亲家庭等）
- `教育背景`：12种选项（普通公立、学生会干部、转校生等）
- `兴趣偏好`：30种选项（阅读写作、运动、音乐等）
- `价值观体系`：15种选项（重视真诚、追求成就等）
- `角色缺陷`：10种选项（过于敏感、缺乏自信等）
- `讨厌的事情`：20种选项（背叛欺骗、失败被否定等）
- `人际关系与信任风格`：15种选项（主动热情、慢热型等）
- `爱情风格`：10种选项（浪漫主义、理性务实等）
- `故事及剧情发展曲线`：8种选项（稳步上升、波动型等）

**数据来源**：`backend/data/character_attributes.py`

---

#### 3. `character_states` 表（角色状态值表）

**用途**：存储角色的动态状态值（好感度、情绪等）

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
|--------|------|------|--------|------|
| `id` | INTEGER | PK, AUTO_INCREMENT | - | 记录ID |
| `character_id` | INTEGER | FK → characters.id | - | 角色ID（外键） |
| `favorability` | FLOAT | - | 0.0 | 好感度（-100到100） |
| `trust` | FLOAT | - | 0.0 | 信任度（0-100） |
| `hostility` | FLOAT | - | 0.0 | 敌意（0-100） |
| `dependence` | FLOAT | - | 0.0 | 依赖度（0-100） |
| `emotion` | FLOAT | - | 50.0 | 情绪（0-100） |
| `stress` | FLOAT | - | 0.0 | 压力（0-100） |
| `anxiety` | FLOAT | - | 0.0 | 焦虑（0-100） |
| `happiness` | FLOAT | - | 50.0 | 快乐（0-100） |
| `sadness` | FLOAT | - | 0.0 | 悲伤（0-100） |
| `confidence` | FLOAT | - | 50.0 | 自信度（0-100） |
| `initiative` | FLOAT | - | 50.0 | 主动度（0-100） |
| `caution` | FLOAT | - | 50.0 | 谨慎度（0-100） |
| `updated_at` | DATETIME | - | NOW() | 更新时间 |

**状态值说明**：
- **好感度（favorability）**：-100到100，负数表示反感
- **其他状态值**：0-100，数值越高表示该状态越强烈
- **默认值**：大部分状态默认50（中性），好感度默认0（中性）

## 🔍 第二部分：ChromaDB向量数据库设计

### 数据库配置
- **类型**：ChromaDB（基于SQLite的持久化存储）
- **存储路径**：`项目根目录/vector_db/`
- **数据库文件**：`vector_db/chroma.sqlite3`
- **Embedding模型**：`text2vec-chinese`（768维向量）
- **配置位置**：`backend/config.py`

### ChromaDB物理结构（SQLite表）

ChromaDB使用SQLite作为底层存储，包含以下核心表：

#### 核心表说明

| 表名 | 用途 |
|------|------|
| `collections` | 存储集合（collection）信息 |
| `collection_metadata` | 存储集合的元数据 |
| `embeddings` | 存储向量embedding的索引信息 |
| `embedding_metadata` | 存储每个文档的metadata |
| `embeddings_queue` | 写入队列（ChromaDB内部使用） |
| `segments` | 向量分段信息（用于索引优化） |
| `segment_metadata` | 分段元数据 |
| `embedding_fulltext_search_*` | 全文检索相关表（FTS） |
| `databases` / `tenants` | 多租户支持 |
| `migrations` | 数据库迁移版本 |

---

### 应用层设计（本项目）

#### Collection：`story_events`

**集合名称**：`story_events`  
**用途**：存储所有游戏事件和对话数据  
**Embedding模型**：`text2vec-chinese`（768维）  
**Collection Metadata**：
```json
{
  "description": "存储剧情事件内容",
  "embedding_model": "text2vec-chinese"
}
```

---

#### 文档ID格式（Document IDs）

**完整事件**：
```
{character_id}_{event_id}
示例：15_borrow_pen, 16_cafeteria_seat_share
```

**对话轮次**：
```
{character_id}_{event_id}_round_{dialogue_round}
示例：15_borrow_pen_round_1, 16_cafeteria_seat_share_round_5
```

---

#### Documents（文档内容 - 会被向量化）

**完整事件的Documents格式**：
```
{story_background}

{dialogue_text}

其中：
- story_background: 故事背景文本
- dialogue_text: 所有对话的组合（格式：角色: xxx\n玩家: xxx）
```

**对话轮次的Documents格式**：
```
{story_background}

第{dialogue_round}轮对话：
角色: {character_dialogue}
玩家: {player_choice}
```

**注意**：Documents中的文本会被embedding模型转换为768维向量，用于语义相似度检索。

---

#### Metadatas（元数据 - 用于过滤，不会被向量化）

**完整事件的Metadata**：
```python
{
    'character_id': str,           # 角色ID（用于过滤）
    'event_id': str,               # 事件ID
    'type': 'complete_event',      # 类型标记
    'dialogue_rounds': str,        # 对话轮次数
    'event_context': str,          # 事件上下文描述
    'title': str,                  # 事件标题
    # 注意：当前未包含情绪相关字段
}
```

**对话轮次的Metadata**：
```python
{
    'character_id': str,           # 角色ID（用于过滤）
    'event_id': str,               # 事件ID
    'type': 'dialogue_round',      # 类型标记
    'dialogue_round': str,         # 对话轮次编号
    'event_context': str,          # 事件上下文描述
    'title': str,                  # 事件标题
    'scene': str,                  # 场景ID
    # 注意：当前未包含情绪相关字段
}
```

**当前Metadata字段列表**：
- `character_id`：角色ID
- `event_id`：事件ID
- `type`：类型（`complete_event` 或 `dialogue_round`）
- `dialogue_round`：对话轮次（仅对话轮次有）
- `dialogue_rounds`：对话轮次数（仅完整事件有）
- `event_context`：事件上下文描述
- `title`：事件标题
- `scene`：场景ID（仅对话轮次有）

**缺失字段**（建议添加）：
- `emotion`：情绪值
- `favorability`：好感度
- `trust`：信任度
- `hostility`：敌意
- `emotion_level`：情绪级别（high/medium/low）

## 🔗 数据库关联关系

### PostgreSQL ↔ ChromaDB 关联

```
PostgreSQL (结构化数据)
├── characters.id
│   └── character_states (状态值)
│       └── emotion, favorability, trust...
│
└── ChromaDB (向量数据)
    └── story_events collection
        ├── metadata.character_id → characters.id
        ├── documents (包含对话和事件文本)
        └── metadata (包含事件元信息)
```

**关联方式**：
- 通过 `character_id` 关联
- PostgreSQL存储角色的静态属性和动态状态
- ChromaDB存储角色的历史事件和对话（用于语义检索）

## 📝 数据流程

### 1. 角色创建流程

```
用户输入角色信息
    ↓
PostgreSQL: 创建 Character 记录
    ↓
PostgreSQL: 创建 CharacterAttribute 记录（多个）
    ↓
PostgreSQL: 创建 CharacterState 记录（初始化状态值）
    ↓
返回 character_id
```

### 2. 事件生成和存储流程

```
游戏进行中
    ↓
生成事件（story_engine.py）
    ↓
每轮对话后：
    ├── PostgreSQL: 更新 CharacterState（状态值变化）
    └── ChromaDB: 保存对话轮次（save_dialogue_round_to_vector_db）
        ├── document: 故事背景 + 对话内容
        └── metadata: 事件信息
    ↓
事件结束后：
    └── ChromaDB: 保存完整事件（save_event_to_vector_db）
        ├── document: 故事背景 + 所有对话
        └── metadata: 事件信息
```

### 3. 事件检索流程

```
需要生成新事件
    ↓
ChromaDB: 检索相似历史事件（search_similar_events）
    ├── 基于query文本进行语义检索
    ├── 使用where条件过滤（character_id等）
    └── 返回相似的历史事件documents
    ↓
AI生成器: 基于历史事件生成新事件
    ↓
PostgreSQL: 获取角色当前状态值
    ↓
生成新的事件和对话
```

## 🎯 数据存储位置总结

| 数据类型 | 存储位置 | 存储格式 | 用途 |
|---------|---------|---------|------|
| 角色基本信息 | PostgreSQL (`characters`) | 结构化表 | 角色管理 |
| 角色决定因素 | PostgreSQL (`character_attributes`) | 结构化表 | 角色属性查询 |
| 角色状态值 | PostgreSQL (`character_states`) | 结构化表 | 状态值查询和更新 |
| 事件和对话文本 | ChromaDB (`story_events`) | 向量化文档 | 语义检索、历史参考 |
| 事件元数据 | ChromaDB (`story_events.metadata`) | 键值对 | 精确过滤、快速查询 |

## ⚠️ 当前设计问题

### 1. 情绪数据未向量化
- **问题**：情绪等状态值未存储到ChromaDB
- **影响**：无法基于情绪状态检索相似历史事件
- **建议**：参考"情绪向量化策略分析.md"

### 2. Metadata字段不完整
- **问题**：缺少情绪相关字段
- **影响**：无法按情绪过滤数据
- **建议**：添加情绪字段到metadata

### 3. Documents未包含情绪信息
- **问题**：documents文本中未包含情绪标签
- **影响**：无法进行情绪相关的语义检索
- **建议**：添加情绪语义标签到documents

## 🔧 数据库操作接口

### PostgreSQL操作（DatabaseManager）

**主要方法**：
- `create_character()`：创建角色
- `get_character()`：获取角色信息
- `get_character_states()`：获取角色状态值
- `update_character_states()`：更新角色状态值
- `get_character_attributes()`：获取角色属性

### ChromaDB操作（VectorDatabase）

**主要方法**：
- `add_event()`：添加完整事件
- `add_dialogue_round()`：添加对话轮次
- `search_similar_events()`：搜索相似历史事件
- `search_recent_dialogues()`：搜索最近对话
- `get_event_by_id()`：根据ID获取事件

## 🎨 设计特点

1. **双数据库架构**：结构化数据用PostgreSQL，非结构化文本用ChromaDB
2. **向量化检索**：支持基于语义的相似事件检索
3. **状态值管理**：PostgreSQL存储实时状态，ChromaDB存储历史记录
4. **关联设计**：通过character_id关联两个数据库
5. **扩展性**：易于添加新的状态值和属性类型
